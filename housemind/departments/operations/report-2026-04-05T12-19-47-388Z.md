# **Operations Department Report: MVP Build Review**

## **Executive Summary**
The MVP build is a strong foundational step for the HouseMind platform, but several operational gaps need addressing to streamline workflows and ensure scalability. Key areas requiring improvement include catalog management, invite flow transparency, onboarding tracking, and operational visibility. Addressing these gaps requires additional admin tools, backend event tracking, and collaboration with the Tech department to support our architect-first strategy effectively.

---

## **Key Decisions Made**
1. A lightweight admin product management interface will be requested to facilitate product curation, dynamic category updates, and product labeling without requiring developer support.
2. An admin invite dashboard will be developed to improve the visibility and management of invite tokens and ensure smooth architect onboarding.
3. Backend event logging for critical onboarding milestones (account creation, first board, and first share) will be implemented to track architect engagement through the onboarding funnel.
4. A new database table will track "Request Availability" actions, with auto-generated reports and email notifications for operations to facilitate supplier recruitment.
5. Dashboards for Operational Visibility:
   - Onboarding Funnel Dashboard to track architect progress through the onboarding checklist.
   - User Activity Dashboard to monitor architect engagement and "Request Availability" actions.

---

### **2. Admin Invite Dashboard**

### **3. Onboarding Event Tracking**
- **Description:** Implement backend logging for onboarding milestones and use this data to build an onboarding funnel dashboard.
- **Tracked Events:**
  - Account created
  - First project/board created
  - First share made
- **Owner:** Tech (Backend Team)

- **Operations**: Use the generated data to monitor funnel performance and guide feedback calls.

---

### **4. "Request Availability" Logging & Notifications**
- **Description:** Extend the "Request Availability" feature to include backend tracking of user actions with reporting and notification capabilities.
- **Owner:** Tech (Backend & Frontend Teams)
- **Dependencies:**
  - New `availability_requests` table to log user actions.
  - Backend changes to record requests and link them to user/product IDs.
  - Frontend: Update the current "Request Availability" button to hook into updated backend functionality.
  - Email notification system for admins to receive updates on new availability requests.

---

### **5. User Activity and Onboarding Dashboards**
- **Description:** Build operational dashboards to monitor architect activity and track onboarding progress.
  - **User Activity Dashboard:** Number of pinned products, shared boards, active architects, and availability requests.

## **Risks and Open Questions for the Product Team**
1. **Scalability of the Catalog:**
   - The current catalog's static category structure isn’t optimized for scaling to 100-150 products or changing product categories. If the scope grows (e.g., more categories/product types), how will this be handled in future phases?
2. **Onboarding Drop-offs:**
   - If the onboarding flow data shows significant drop-offs (e.g., architects not creating boards or sharing projects), should we consider redesigning the UX or adding gamification? Would PM initiate qualitative testing or surveys for dropped-out users?
3. **Feedback Flow Integration:**
   - Would you prefer feedback collection to remain anonymous at the MVP stage, or should it be integrated with user actions to provide more actionable insights for operations?
4. **Timeline for Enhancements:**
   - With limited Tech resources, which deliverables are feasible in Wave 2 of the MVP? If some recommendations are deprioritized, how might this impact operations and the architect onboarding experience?
