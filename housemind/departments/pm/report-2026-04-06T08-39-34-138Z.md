## CRITICAL CONFLICTS & DECISIONS NEEDED FROM FOUNDER

### 🚨 DECISION 1: Resource Allocation Crisis
**Conflict:** Tech is laser-focused on the `/projects/[id]` god page (60% of app complexity), but Operations needs 5 admin tools and Marketing needs onboarding features. All three departments assume their requests are Wave 1.

**Founder must decide:**
- Does Tech build ONLY the core board experience in Wave 1, or do we split resources?
- If split: What's the minimum viable admin tooling to launch beta?
- Can we launch beta WITHOUT onboarding checklists/walkthroughs/drip emails?

**PM Recommendation:** Ship board experience + bare-minimum invite dashboard. Defer onboarding automation and advanced admin tools to Wave 2. Co-owner does manual onboarding calls with first 5 architects.

### 🚨 DECISION 2: Catalog Quality vs. Speed
**Conflict:** Marketing flags that 20 seed products with potentially low-quality images won't impress architects. Operations asks if catalog is "launch-ready." Tech treated this as demo data, not production content.

**Founder must decide:**
- Do we delay beta to curate 100-150 high-quality products (Ops estimate: 3-4 weeks)?
- Or launch with 40-60 "good enough" products and iterate based on architect feedback?
- Who sources product images/specs: Ops manually, or do we need supplier partnerships first?

**PM Recommendation:** Launch with 60 curated products (15 per category). Co-owner leverages existing supplier relationships to get hi-res images + specs. Ops owns curation. 2-week parallel track while Tech builds.

## UNIFIED PRIORITIES (Forced-Rank Order)

### **PRIORITY 1: Ship Core Board Experience** (Weeks 1-6)
**Owner:** Tech  
**Outcome:** `/projects/[id]` page functional with image carousel, annotations, product linking

**Architecture Decision (PM):** 
- **Option D (Progressive Disclosure)** with modifications:
  - Single URL, board view default
  - Tap image → full-screen annotation mode (modal overlay)
  - Zustand for state (lighter than Context for mobile performance)
  - Component tree:
    ```
    ProjectPage/
      ├─ BoardView (default)
      ├─ AnnotationModal (overlay, activated by image tap)
      │   ├─ ImageCarousel
      │   ├─ AnnotationLayer (pins + circular menu)
      │   └─ ProductPanel (filtered)
      └─ SharedState (Zustand store)
    ```

**Why:** Matches "one workspace" constraint, keeps testing simple (modal is isolated component), avoids sub-route fragmentation.

**Tech Deliverables:**
- Image carousel with lazy loading (Week 1-2)
- Annotation layer with long-press + emoji pins (Week 2-4)
- Product panel with filtering (Week 3-4)
- Zustand state + Cypress E2E tests (Week 5-6)

**Dependencies:**
- **NONE** (self-contained, assuming catalog API exists)
