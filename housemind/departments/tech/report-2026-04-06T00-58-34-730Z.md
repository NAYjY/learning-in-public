# HouseMind Technology Department Report  
## Solving the `/projects/[id]` "God Page" Problem

---

### **Executive Summary**  
The `/projects/[id]` page will be designed as a single cohesive workspace that incorporates progressive disclosure techniques to manage complexity and ensure a seamless mobile-first experience. We will implement a modular architecture, leveraging **Zustand** for lightweight global state management and reusable React components for image carousels, annotations, and product linking. This approach ensures improved maintainability, clear state synchronization, and optimized performance for mid-range mobile devices (e.g., Samsung A series, iPhone SE).

---

### **Key Decisions Made**  

1. **Component Hierarchy:** The page will be divided into modular components: `ImageCarousel`, `AnnotationLayer`, `Pin`, `CircularMenu`, and `ProductPanel`, ensuring decoupled logic and clear responsibilities.
2. **State Management:** Use **Zustand** for global state to handle shared properties like `activeImageIndex`, `annotations`, and `productFilter`, paired with local state for lightweight per-component interactions.
3. **Progressive Disclosure:** No sub-routes—use a single URL (`/projects/[id]`) with modal-based flows for annotation mode and product linking while maintaining the feel of a single workspace.
4. **Mobile Interaction Handling:** Implement long-press and drag gestures with the `react-use-gesture` library, with pinned annotations rendered at relative `x, y` positions on images.
5. **Performance Optimization:** Lazy-load images and annotations using **Intersection Observers**, debounce state updates, and precompress images on S3 upload to ensure smooth operation on low-end mobile devices.

### **Risks and Open Questions for PM**

1. **Gesture Ambiguity:** Long-press can often conflict with default browser behavior (context menus) or may be inconsistent across devices. We’ll need final confirmation on whether this interaction is critical or should be complemented with a fallback (e.g., a separate "Add Annotation" button).
2. **Annotation Overlap:** UX for managing overlapping annotations in a single image, especially at smaller screen sizes, is currently unclear. Should annotations auto-adjust to avoid collisions, or should overlapping pins be allowed?
3. **Annotation Limits:** Is there a maximum number of annotations or products we expect per image? This will help us scope performance optimizations accordingly.
4. **Undo/Redo UX Design:** What are the exact undo/redo behaviors that users (architects/contractors) expect? For instance, should this apply to all actions (product linking, annotation deletion) or only specific types of actions?
5. **Offline Support:** Should users be able to annotate and link products while offline, with state synced back to the server later? (If yes, this introduces significant complexity.)
